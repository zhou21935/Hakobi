- [x] 1.2 依需求「Orders store rejects invalid data on write」與「Order validation rules have a single shared source」，依設計決策「驗證與整理邏輯放在新的 src/domain/orderValidation.js 純函式模組」，於 `src/domain/orderValidation.js` 實作 `validateOrder(data)`，回傳 `{ isValid, errors: { name, amount, productCategories } }`：`name` trim 後為空字串時 `errors.name = '商品名稱不可為空'`；`amount` 非大於 0 的有限數字時 `errors.amount = '金額須為大於 0 的數字'`；`productCategories` 不是陣列或長度為 0 時 `errors.productCategories = '請至少選擇一項商品分類'`；三者皆為 `null` 時 `isValid` 為 `true`；驗證：`orderValidation.spec.js` 新增測試涵蓋合法資料（`isValid: true`，三個 errors 皆為 `null`）與三種個別不合法情境（空名稱、金額 0/負數、空商品分類陣列），每種情境斷言對應的 errors 欄位為指定的錯誤訊息字串，其餘 errors 欄位為 `null`

## 2. Store 套用共用驗證

- [x] 2.1 依需求「Adding an order with invalid data is rejected」、「Adding an order with valid data succeeds」與需求「User can create a preorder order with required field validation」，依設計決策「stores/orders.js 的 addOrder / updateOrder 在寫入前呼叫 normalizeOrderInput 與 validateOrder，失敗回傳 null」，修改 `src/stores/orders.js` 的 `addOrder(orderData)`：呼叫 `normalizeOrderInput` 整理資料後呼叫 `validateOrder`，`isValid` 為 `false` 時直接 `return null` 且不執行 `orders.value.push(...)`；合法時使用整理後的 `name`/`amount` 建立訂單並維持既有欄位預設值邏輯；驗證：`src/stores/__tests__/orders.spec.js` 新增測試，`store.addOrder({ name: '', amount: 10, productCategories: ['merch'] })`、`store.addOrder({ name: 'test', amount: 0, productCategories: ['merch'] })`、`store.addOrder({ name: 'test', amount: 10, productCategories: [] })` 皆回傳 `null` 且 `store.orders` 長度不變；`store.addOrder({ name: '  test  ', amount: '10', productCategories: ['merch'] })` 回傳的訂單 `name` 為 `'test'`、`amount` 為 `10`
- [x] 2.2 依需求「Updating an order with invalid data is rejected」與需求「Orders can be tagged with one or more product categories」，修改 `src/stores/orders.js` 的 `updateOrder(id, orderData)`：找到既有訂單後，先計算 `merged = { ...orders.value[index], ...orderData }`，對 `merged` 呼叫 `normalizeOrderInput` 與 `validateOrder`，`isValid` 為 `false` 時直接 `return null` 且不修改 `orders.value[index]`；合法時將整理後的 `merged` 寫入 `orders.value[index]` 並回傳；驗證：`orders.spec.js` 新增測試，先用合法資料 `addOrder` 建立一筆訂單，再呼叫 `store.updateOrder(id, { name: '' })` 斷言回傳 `null` 且該筆訂單在 `store.orders` 中的 `name` 維持原值不變；再呼叫 `store.updateOrder(id, { isPaid: true })`（不含 name/amount/productCategories 的局部更新）斷言正常成功、回傳的訂單 `isPaid` 為 `true` 且 `name`/`amount`/`productCategories` 維持原值
- [x] 2.3 更新 `src/stores/__tests__/orders.spec.js` 內既有的 4 個測試（`addOrder isPreorder` 與 `addOrder productCategories` 兩個 describe 內、目前呼叫 `store.addOrder({ name: 'test', amount: 10 })` 未帶 `productCategories` 的測試），改為傳入 `productCategories: ['merch']`（或該測試本身已在測試的分類陣列），確保這些測試在新規則下仍呼叫合法資料並繼續驗證原本的行為（`isPreorder` 預設值、`productCategories` 預設值與自訂值）；驗證：`npx vitest run src/stores/__tests__/orders.spec.js` 全數通過
